Vamos continuar a FASE 1 (infraestrutura) do sistema de gestão de matrículas acadêmicas,
seguindo o fluxo definido em CLAUDE.md. Esta etapa adiciona a stack de observabilidade —
ainda nenhum código de domínio deve ser criado.

ESCOPO DESTA ETAPA:

1. Docker Compose — adicionar:
   - Prometheus (scrape do endpoint de métricas do Spring Boot)
   - Grafana (com provisionamento automático de datasources: Prometheus, Jaeger e o backend
     de logs escolhido — não pedir para eu configurar manualmente na UI)
   - Jaeger (aceitando OTLP, não o formato legado) para rastreamento distribuído
   - Um backend de log centralizado que permita visualizar logs de todos os containers sem
     precisar rodar `docker logs`/entrar no container — avalie Grafana Loki + Promtail como
     default (justificativa: integra nativamente ao Grafana já presente na stack, permite
     correlacionar log ↔ trace ↔ métrica pelo mesmo traceId na mesma tela) vs. alternativas
     mais pesadas (EFK/ELK). Registre a escolha em docs/DECISIONS.md antes de implementar.

2. Spring Boot — configurar:
   - Actuator com health checks (`/actuator/health`) já exigidos pelo PRD.
   - Micrometer + `micrometer-registry-prometheus` para expor métricas básicas em
     `/actuator/prometheus`.
   - Micrometer Tracing com bridge para OpenTelemetry, exportando via OTLP para o Jaeger.
   - Correlation ID / trace ID nos logs: garantir que traceId/spanId apareçam
     automaticamente em cada linha de log (via MDC do Micrometer Tracing) e que o log saia
     em formato estruturado (JSON) compatível com o backend de log escolhido no item 1 —
     não é aceitável ter trace ID no Jaeger sem esse mesmo ID aparecer no log correspondente.

3. Documente no compose.yaml comentários explicando o papel de cada serviço novo — o
   compose já vai estar com bastante peso (postgres, redis, rabbitmq, keycloak +
   observabilidade), então avalie e proponha se faz sentido usar Docker Compose profiles
   (ex: `--profile observability`) para tornar a stack de observabilidade opcional na
   subida padrão, mantendo o ciclo de desenvolvimento do dia a dia leve, e documentar isso
   como decisão consciente (trade-off entre "tudo sempre de pé" vs. "observabilidade sob
   demanda").

FORA DO ESCOPO desta etapa: instrumentação de trace propagando através das mensagens do
RabbitMQ (isso fica para quando o producer/consumer de eventos existir de fato — registre
como item pendente na spec, não implemente agora).
