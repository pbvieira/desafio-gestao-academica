1. Documento arquitetural curto (PRD §04 — deliverable distinto do README)
   - O PRD pede explicitamente uma "documentação arquitetural curta explicando decisões,
     trade-offs, riscos conhecidos, evolução da solução e tratamento de falhas na
     mensageria" — isso NÃO existe hoje como arquivo próprio (só `docs/DECISIONS.md`, que
     é a fonte granular de ~50 decisões, e o README, que é operacional/instruções).
   - Crie `docs/ARQUITETURA.md`: um documento curto (não é para reescrever as decisões,
     é para contar a história arquitetural em poucas páginas navegáveis, com link pontual
     para entradas específicas de DECISIONS.md quando precisar de profundidade). Estrutura
     sugerida:
     - Visão geral do desenho (monólito modular Spring Modulith, contexto acadêmico vs.
       secundário) e por quê (D001).
     - As 3-5 decisões mais defensáveis/questionáveis do projeto, com o trade-off real
       (ex: D001 monólito modular vs. microsserviços, D024 estratégia de concorrência com
       números reais da Fase 6, D002 RabbitMQ vs. Kafka).
     - **Seção dedicada: como o padrão Outbox foi implementado via Spring Modulith.**
       Motivação explícita do Pablo: quem não conhece Spring Modulith NÃO vai enxergar o
       outbox lendo o código, porque não existe uma classe `OutboxPublisher` nem uma
       tabela `outbox` criada à mão — o padrão está implementado pela infraestrutura do
       framework (Event Publication Registry). A seção precisa tornar isso visível para o
       avaliador:
       - O conceito em uma frase: por que outbox existe (atomicidade entre a mudança de
         estado no banco e a publicação do evento — sem "dual write" para o broker).
       - O fluxo concreto neste projeto, passo a passo: evento de domínio publicado
         dentro da MESMA transação do caso de uso → Modulith persiste a publicação na
         tabela `event_publication` (mostrar o schema/migration Flyway correspondente)
         → após o commit, o evento é externalizado para o RabbitMQ (anotação
         `@Externalized` ou configuração equivalente usada no projeto) → publicação
         marcada como completa; se a entrega falhar, a publicação fica incompleta e é
         reprocessada (explicar a política de republicação configurada:
         `republish-outstanding-events-on-restart` ou o que estiver de fato em uso).
       - Mapa "onde está no código": apontar arquivo/classe/anotação de cada peça do
         fluxo (evento, publicação, listener/externalização, configuração, migration),
         para o avaliador conseguir seguir a trilha sozinho.
       - Um diagrama simples do fluxo (Mermaid ou ASCII) vale muito aqui.
       - O que ganhamos de graça vs. um outbox artesanal, e as limitações aceitas
         (ex: polling/republicação em restart vs. relay dedicado tipo Debezium).
       - Linkar a(s) entrada(s) de DECISIONS.md que originaram essa escolha.
     - Riscos conhecidos aceitos (ex: D035 rede do compose sem mTLS, qualquer outro achado
       de security-auditor mantido deliberadamente).
     - Caminho de evolução: como cada módulo se extrairia como serviço separado se
       necessário (pergunta explícita do PRD/entrevista).
     - Tratamento de falhas de mensageria (retry/DLQ/outbox — já implementado, resumir e
       cruzar com a seção de outbox acima, sem duplicar).
   - Sugestão adicional: avaliar gerar o diagrama de módulos com o `Documenter` do próprio
     Spring Modulith (saída PlantUML/C4 a partir dos testes de verificação de módulos) e
     incluí-lo/referenciá-lo em ARQUITETURA.md — é evidência automática de que a
     modularização declarada é a real.

2. Seção "Observabilidade" no README — papel de cada software da stack
   - Hoje o README (presumivelmente) só instrui a subir o compose. Falta explicar O QUE é
     cada componente de observabilidade e COMO ele funciona, de forma sucinta (2-4 frases
     por componente), para um avaliador que não conhece a stack.
   - Para CADA software de observabilidade presente no docker-compose (usar o que existe
     de fato no compose — ex.: Micrometer/OpenTelemetry no app, Prometheus, Grafana,
     Loki/Promtail, Tempo ou Zipkin/Jaeger, OTel Collector se houver), documentar:
     - Papel: que sinal ele cobre (métricas, logs ou traces) e o que faz na arquitetura.
     - Como funciona, em linguagem direta: ex. "Prometheus opera em modelo pull — faz
       scrape periódico do endpoint `/actuator/prometheus` exposto pelo Micrometer e
       armazena as séries temporais"; "Grafana não armazena nada — só consulta as
       datasources (Prometheus/Loki/Tempo) e renderiza dashboards"; "o trace é propagado
       entre HTTP e RabbitMQ via headers de contexto (traceparent), o que permite ver a
       matrícula ponta-a-ponta num único trace".
     - Como os componentes se conectam entre si (um mini-diagrama ou lista de setas
       app → collector → backend → Grafana resolve).
     - URL local e credenciais de cada UI, e 1 exemplo de "o que olhar primeiro" em cada
       uma (ex: dashboard X no Grafana, query de log de uma matrícula no Loki, trace de
       um POST /matriculas no Tempo).
   - Se ficar longo demais para o README, extrair para `docs/OBSERVABILIDADE.md` e deixar
     no README um resumo de 1 parágrafo + link — decidir pelo tamanho real.
